# Use remote Terraform plans for PR validation

Pull requests targeting `main` will run Terraform plans for both `dev` and `prd` against the remote Azure state backends using OIDC-authenticated credentials, because plan-aware security and cost checks need evaluated Terraform plans for every environment rather than raw HCL alone. PR validation is split into fast Terraform hygiene checks, explicit GitHub Actions security scanning with zizmor, and slower plan-aware Checkov checks so formatting, linting, and workflow-security failures stop before Azure-authenticated planning. The plan-aware job only runs after hygiene and workflow-security checks pass, uses GitHub Environments for `dev` and `prd`, and its environment matrix uses `fail-fast: false` so environment failures are reported together. OIDC token permission is granted only to the Azure-authenticated job, preserving least privilege for hygiene and workflow-security jobs. Forked pull requests are excluded from the workflow entirely, because this template is for trusted Sky Haven infrastructure repositories and PR validation depends on protected CI trust boundaries. Third-party GitHub Actions are pinned by full commit SHA, and installed CLI tools are pinned to explicit versions, to reduce supply-chain risk and keep validation deterministic. Validation tool configuration is centralised under `.github/validation/`, replacing the old Super-Linter-oriented `.github/linters/` convention. The workflow reuses the existing state-container creation and lease-breaking actions so PR validation behaves consistently with Terraform runs. Checkov will run plan-aware deep analysis using both the Terraform plan JSON and source Terraform so it can combine evaluated resource changes with source-level context. Terraform plan files and plan JSON are generated only in the ephemeral runner workspace and are not uploaded as artifacts by default, because plans may expose sensitive values or infrastructure details. All Checkov findings are treated as merge-blocking unless explicitly suppressed in the central Checkov configuration; findings fail the workflow logs and are not uploaded as SARIF. Infracost runs only after Checkov passes, fails if its API key is missing, and updates separate sticky, comment-only cost summaries for each environment so reviewers see clean `dev` and `prd` deltas without noisy comments from failed intermediate revisions. Documentation-only and README-only pull requests are excluded from Terraform PR validation to avoid unnecessary Azure-authenticated plans.
